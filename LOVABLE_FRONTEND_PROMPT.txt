2. HARD CONSTRAINTS
==================================================

Обязательные ограничения:

1. Не менять backend.
2. Не менять существующие endpoint paths.
3. Не менять существующие frontend routes.
4. Все fetch-запросы должны идти с `credentials: "include"`.
5. Все POST/PUT/DELETE должны сначала иметь CSRF token через `GET /api/auth/csrf/` и передавать `X-CSRFToken`.
6. Auth state определяется через `GET /api/auth/session/`.
7. Feature gating делается по `user.features`.

10. Терминал должен остаться полноценным сценарным экраном, а не декоративным компонентом.

==================================================
5. AUTH AND SESSION CONTRACT
==================================================

Использовать такие endpoint'ы:

- `GET /api/auth/session/`
  Возвращает:
  - `{ authenticated: false, user: null }`
  - или `{ authenticated: true, user: { id, username, email, is_staff, features } }`

- `GET /api/auth/csrf/`
  Возвращает `{ csrfToken }`

- `GET /api/auth/ws-token/`
  Возвращает `{ token }`

- `POST /api/auth/login/`
  Body:
  - `{ username, password, auth_mode }`
  Где `auth_mode` обычно `"auto"` или `"local"`

- `POST /api/auth/logout/`

Feature flags внутри `user.features`:
- `servers`
- `dashboard`
- `agents`
- `studio`
- `settings`
- `orchestrator`

Важно:
- `dashboard` доступен staff-пользователям с agent feature;
- `studio` и `agents` gated через backend permissions;
- при отсутствии доступа route должен redirect'ить на `/servers`.

6. BACKEND BASE URLS AND DEV BEHAVIOR
==================================================

Локальная разработка:
- Django обычно на `http://127.0.0.1:9000`
- frontend dev обычно на `http://127.0.0.1:8080`

Важно сохранить совместимость с текущим подходом:
- `VITE_API_BASE` может быть пустым;
- `VITE_BACKEND_ORIGIN` может использоваться для direct backend links;
- `VITE_DJANGO_WS_URL` и `VITE_WS_HOST` могут использоваться для websocket base;
- в dev websocket cookie может не проксироваться, поэтому нужен `ws_token`.

11. SPECIAL RDP/TERMINAL NOTE
==================================================

Есть важная особенность:
- frontend route для RDP: `/servers/:id/rdp`
- но direct backend path helper сейчас использует `/servers/<id>/terminal/`
- backend сам уже решает, отдать SSH terminal template или RDP template

Поэтому:
- не выдумывай отдельный backend endpoint `/servers/<id>/rdp/`, если он не нужен;
- сохрани frontend route `/servers/:id/rdp`, но уважай существующий backend flow;
- terminal/RDP UX должен быть аккуратным и безопасным.
